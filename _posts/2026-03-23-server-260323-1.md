## 1. CGI와 FastCGI

- **CGI (Common Gateway Interface)**
웹 서버가 외부 프로그램(PHP, Python 등)을 실행하고 결과를 클라이언트로 전달하는 규칙.
- **CGI 특징**
    - 요청마다 새 프로세스 생성 → 성능 저하, 서버 부담 ↑
    - 장점: 구조 단순, 다양한 언어 연결 가능
    - 단점: 요청 많으면 느림
- **FastCGI**
    - CGI의 개선판
    - 미리 생성된 프로세스 풀을 사용 → 요청마다 새 프로세스 생성하지 않음
    - 성능 향상, 리소스 효율적
- **웹 서버와 프로그램 간 데이터 통신**
    - 같은 서버 안에서 소켓/IPC로 통신
    - 다른 서버를 거치지 않고 PHP-FPM과 웹 서버 간 통신 가능

---

## 2. PHP-FPM (PHP FastCGI Process Manager)

- **정의:** PHP 코드를 실행하고 PHP 프로세스 풀을 효율적으로 관리하는 관리자
- **역할**
    1. PHP 코드 실행
    2. PHP 프로세스 풀 관리
    3. 멀티 PHP 버전 지원
    4. 프로세스/사용자/권한/모듈 독립 관리
- **동작 구조**

```
[웹 브라우저 요청]
        |
        v
[웹 서버: Apache/Nginx]
        |
        v  (FastCGI 소켓)
[PHP-FPM 서비스 (데몬)]
        |
        v
[PHP-FPM 풀: PHP 프로세스 모음]
        |
        v
[PHP 코드 실행 → 결과 반환]
```

## 3. PHP-FPM 서비스, 풀, 프로세스 개념

- **서비스 (service)**
OS 단위 실행, PHP-FPM 프로그램 켜고 관리
→ `systemctl start php74-php-fpm`
- **데몬 (daemon)**
백그라운드에서 계속 실행되는 상태
- **풀 (pool)**
요청 처리용 PHP 프로세스 묶음, PHP 버전/모듈/권한별로 독립 관리
- **프로세스 (process)**
풀 안에서 요청을 처리하는 실제 PHP 코드 실행 단위
- **멀티 PHP 환경**
    - PHP 5.6, 7.2, 7.4 → 각 버전별 독립 서비스 + 풀
    - VirtualHost에서 풀 선택 가능

## 4. mod_php와 PHP-FPM 비교

| 항목 | mod_php | PHP-FPM |
| --- | --- | --- |
| 실행 방식 | Apache 프로세스 안에서 직접 PHP 실행 | 별도 PHP-FPM 풀에서 PHP 실행, FastCGI 연결 |
| 성능 | 요청 많으면 Apache 부담 | 요청 많아도 빠름 |
| 멀티 PHP 지원 | 불가 | 가능 |
| 설정 독립성 | 낮음 | 높음 |
| 설치 난이도 | 쉬움 | 조금 복잡 |
| 보안 | 제한적 | 풀 단위 격리 가능 |
- **mod_php 장점:** 설치 간단, 단순 서버에서 충분
- **mod_php 단점:** 단일 PHP, Apache와 함께 실행 → 성능/독립성/보안 제한
- **PHP-FPM 장점:** 멀티 PHP, 성능 좋음, 보안/권한 관리 용이 → 요즘 서버 환경 표준

## 5. VirtualHost와 SetHandler

- **VirtualHost**: 하나의 서버에서 여러 도메인 운영 가능
- **예시 Apache 설정**

```
<VirtualHost *:80>
    ServerName local.knussrp.co.kr
    DocumentRoot /home/html-data/knussrp

    <FilesMatch \.php$>
        SetHandler "proxy:unix:/run/php74-fpm.sock|fcgi://localhost/"
    </FilesMatch>

    DirectoryIndex index.php index.html
    <Directory "/home/html-data/knussrp">
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>
```

- **SetHandler 설명**
    - Apache가 `.php` 요청을 PHP-FPM 풀로 전달
    - `proxy` → mod_proxy 사용
    - `unix:/run/php74-fpm.sock` → 같은 서버 내 소켓
    - `fcgi://localhost/` → FastCGI 프로토콜, localhost = 같은 서버

## 6. UNIX 소켓 vs TCP

| 구분 | 연결 방식 | 장점 | 단점 | 사용 케이스 |
| --- | --- | --- | --- | --- |
| UNIX 소켓 | 파일 기반 | 빠름, OS 권한 관리 가능 | 같은 서버에서만 사용 | 대부분 서버 환경 |
| TCP | IP/포트 기반 | 원격 서버 가능, 컨테이너 환경 | TCP 오버헤드 존재 | Docker, 원격 FPM, 로드밸런싱 |
- 대부분 환경에서는 **UNIX 소켓 사용**
- TCP는 특수 상황에서만 필요

---

## 7. PHP 설정 파일 정리 (php.ini / [www.conf](http://www.conf/))

| 파일 | 적용 대상 | 역할 |
| --- | --- | --- |
| php.ini | PHP 전체 설정 | 확장 모듈, 기본 옵션, CLI 포함 |
| [www.conf](http://www.conf/) | PHP-FPM 풀 단위 | 사용자, 권한, 프로세스 수, 풀 단위 옵션 |
- FPM은 php.ini 기본 적용 후, www.conf에서 풀 단위로 오버라이드 가능
- mod_php는 php.ini만 사용
- CLI, 크론용 PHP는 별도 php.ini 사용

---

## 8. CLI와 크론 작업에서의 PHP

- **CLI (Command Line Interface)**
    - 터미널에서 `$ php script.php` 실행
    - php.ini 적용, FPM과 독립
- **크론 작업**
    - 자동화 스크립트 실행
    - CLI용 PHP 사용, 웹서버 필요 없음

---

💡 **핵심 요점**

1. CGI → 요청마다 프로세스 생성 → 느림
2. FastCGI → 프로세스 재사용 → 성능 ↑
3. PHP-FPM → FastCGI + 프로세스 풀 관리, 멀티 PHP 지원
4. mod_php → Apache 모듈, PHP 엔진 연결, 단일 PHP만 가능
5. VirtualHost + SetHandler → 도메인별 FPM 풀 지정
6. UNIX 소켓 = 같은 서버, TCP = 원격 가능
7. php.ini = 기본 설정, [www.conf](http://www.conf/) = FPM 풀 단위 설정
8. CLI/크론은 php.ini 적용, FPM과 독립
